Hybrid Search: BM25 + Vector Retrieval

Pure vector search misses exact keyword matches. Pure keyword search misses semantic meaning. Hybrid search combines both for significantly better RAG retrieval accuracy.

In This Lesson

  1. Why Pure Vector Search Falls Short
  2. BM25 Keyword Search
  3. EnsembleRetriever in LangChain
  4. Reciprocal Rank Fusion
  5. Hybrid Search with Pinecone
  6. Hands-on Exercise

Why Pure Vector Search Falls Short

Vector search excels at semantic similarity — "car" finds "automobile". But it struggles with exact matches: product codes, error messages, names, and rare technical terms. A user searching for TypeError: 'NoneType' object is not iterable gets better results from keyword search than from an embedding model that may have never seen that exact phrase.

BM25 Keyword Search

  • Great for exact terms, codes, names
  • No GPU required, fast and cheap
  • Misses synonyms and paraphrases
  • Sensitive to typos

Dense Vector Search

  • Understands semantic meaning
  • Works across languages
  • Misses rare exact strings
  • Requires embeddings (cost + latency)
💡 When to use hybrid

Use hybrid search whenever your corpus contains technical content, product identifiers, or user queries that mix natural language with exact strings. In practice, hybrid almost always beats either approach alone.

BM25 Keyword Search

BM25 (Best Match 25) is the standard TF-IDF variant used in Elasticsearch and most search engines. In Python, the rank_bm25 library gives you a standalone retriever.

Python — BM25 retriever setup
# pip install rank_bm25 langchain-community
from langchain_community.retrievers import BM25Retriever
from langchain_core.documents import Document

# Your chunked documents (from text splitting step)
docs = [
    Document(page_content="Python raises TypeError when None is iterable"),
    Document(page_content="Neural networks learn representations via backprop"),
    Document(page_content="BM25 ranks documents by term frequency and inverse document frequency"),
    Document(page_content="Vector embeddings capture semantic similarity between sentences"),
]

# Build BM25 index from documents
bm25_retriever = BM25Retriever.from_documents(docs)
bm25_retriever.k = 3  # return top 3 results

results = bm25_retriever.invoke("TypeError None")
for r in results:
    print(r.page_content)

EnsembleRetriever in LangChain

EnsembleRetriever merges results from multiple retrievers using Reciprocal Rank Fusion (RRF). You set a weight per retriever — typically 0.5/0.5, but tune towards keyword (0.3/0.7) for technical docs or towards vector (0.7/0.3) for conversational queries.

Python — hybrid retriever with EnsembleRetriever
# pip install langchain langchain-openai langchain-community faiss-cpu rank_bm25
from langchain_community.retrievers import BM25Retriever
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings
from langchain.retrievers import EnsembleRetriever
from langchain_core.documents import Document
import os

# Sample corpus
docs = [
    Document(page_content="LangChain is a framework for building LLM applications"),
    Document(page_content="FAISS enables fast approximate nearest-neighbor vector search"),
    Document(page_content="BM25 uses term frequency to rank documents by keyword relevance"),
    Document(page_content="Retrieval-Augmented Generation grounds LLM answers in documents"),
    Document(page_content="OpenAI embeddings convert text into dense vector representations"),
]

# 1. Build BM25 retriever
bm25 = BM25Retriever.from_documents(docs)
bm25.k = 3

# 2. Build FAISS vector retriever
embeddings = OpenAIEmbeddings(api_key=os.environ["OPENAI_API_KEY"])
vectorstore = FAISS.from_documents(docs, embeddings)
faiss_retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# 3. Combine with equal weights (0.5 each, must sum to 1.0)
hybrid = EnsembleRetriever(
    retrievers=[bm25, faiss_retriever],
    weights=[0.5, 0.5]
)

# Query the hybrid retriever
results = hybrid.invoke("how does keyword search rank documents")
print(f"Got {len(results)} results:")
for i, r in enumerate(results, 1):
    print(f"  {i}. {r.page_content[:80]}")
ℹ️ Weight tuning

Start at [0.5, 0.5]. If users search with exact product IDs or error codes → increase BM25 weight to 0.6–0.7. If queries are natural language questions → increase vector weight. Evaluate with a test set of real queries.

Reciprocal Rank Fusion (RRF)

RRF is the algorithm that merges ranked lists from each retriever. For each document, its RRF score is the sum of 1 / (k + rank) across all lists, where k=60 by default. This smoothly blends rankings without needing normalized scores.

Python — manual RRF implementation
from collections import defaultdict

def reciprocal_rank_fusion(ranked_lists: list[list], k: int = 60) -> list:
    """Merge multiple ranked lists using Reciprocal Rank Fusion."""
    scores = defaultdict(float)
    doc_map = {}

    for ranked in ranked_lists:
        for rank, doc in enumerate(ranked, start=1):
            doc_id = doc.page_content  # use content as key
            scores[doc_id] += 1.0 / (k + rank)
            doc_map[doc_id] = doc

    # Sort by descending RRF score
    sorted_ids = sorted(scores, key=lambda x: scores[x], reverse=True)
    return [doc_map[doc_id] for doc_id in sorted_ids]

# Example: merge results from BM25 and FAISS
bm25_results = bm25.invoke("keyword search ranking")
faiss_results = faiss_retriever.invoke("keyword search ranking")
fused = reciprocal_rank_fusion([bm25_results, faiss_results])
print("Fused top result:", fused[0].page_content)

Hybrid Search with Pinecone

Pinecone's serverless index supports hybrid search natively — you send both a dense vector and a sparse BM25 vector in one API call, and Pinecone handles the fusion server-side.

Python — Pinecone hybrid search
# pip install pinecone-client pinecone-text langchain-pinecone
from pinecone_text.sparse import BM25Encoder
from pinecone import Pinecone, ServerlessSpec
from langchain_openai import OpenAIEmbeddings
import os

pc = Pinecone(api_key=os.environ["PINECONE_API_KEY"])

# Create a dotproduct index (required for hybrid)
index_name = "hybrid-rag"
if index_name not in [i.name for i in pc.list_indexes()]:
    pc.create_index(
        name=index_name,
        dimension=1536,
        metric="dotproduct",  # must be dotproduct for hybrid
        spec=ServerlessSpec(cloud="aws", region="us-east-1")
    )

index = pc.Index(index_name)
embeddings = OpenAIEmbeddings(api_key=os.environ["OPENAI_API_KEY"])
bm25_encoder = BM25Encoder().default()  # pre-trained on MS MARCO

corpus = ["RAG grounds LLM responses in retrieved documents",
          "BM25 ranks by term frequency and document frequency"]

# Fit BM25 on your corpus
bm25_encoder.fit(corpus)

# Upsert dense + sparse vectors
for i, text in enumerate(corpus):
    dense = embeddings.embed_query(text)
    sparse = bm25_encoder.encode_documents([text])[0]
    index.upsert(vectors=[{
        "id": str(i),
        "values": dense,
        "sparse_values": sparse,
        "metadata": {"text": text}
    }])

# Query with both dense and sparse vectors
query = "how does keyword relevance work"
dense_q = embeddings.embed_query(query)
sparse_q = bm25_encoder.encode_queries([query])[0]

results = index.query(
    vector=dense_q,
    sparse_vector=sparse_q,
    top_k=3,
    alpha=0.5,  # 0=pure keyword, 1=pure vector
    include_metadata=True
)
for match in results["matches"]:
    print(f"Score {match['score']:.3f}: {match['metadata']['text']}")
⚠️ Pinecone Requirement

Hybrid search requires a dotproduct metric index. Cosine or Euclidean indexes do not support sparse vectors. You cannot change a metric after index creation — plan ahead.

Hands-on Exercise

Build a hybrid retriever over a small technical FAQ and compare results against BM25-only and vector-only retrieval:

Python — exercise: compare retrieval strategies
# Exercise: build three retrievers and evaluate on test queries
from langchain_community.retrievers import BM25Retriever
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings
from langchain.retrievers import EnsembleRetriever
from langchain_core.documents import Document
import os

corpus = [
    Document(page_content="ValueError is raised when a function receives an argument of wrong value"),
    Document(page_content="AttributeError occurs when an attribute reference or assignment fails"),
    Document(page_content="TypeError is raised when an operation is applied to an object of wrong type"),
    Document(page_content="ImportError is raised when an import statement fails to find the module"),
    Document(page_content="KeyError occurs when a dictionary key is not found"),
    Document(page_content="IndexError is raised when a sequence subscript is out of range"),
]

test_queries = [
    "TypeError wrong type",        # favours BM25
    "problem accessing object field", # favours vector
    "KeyError dictionary missing key", # both work
]

# Build retrievers
bm25 = BM25Retriever.from_documents(corpus); bm25.k = 2
vs = FAISS.from_documents(corpus, OpenAIEmbeddings(api_key=os.environ["OPENAI_API_KEY"]))
vec = vs.as_retriever(search_kwargs={"k": 2})
hybrid = EnsembleRetriever(retrievers=[bm25, vec], weights=[0.5, 0.5])

for q in test_queries:
    print(f"\nQuery: {q}")
    print("  BM25:   ", bm25.invoke(q)[0].page_content[:50])
    print("  Vector: ", vec.invoke(q)[0].page_content[:50])
    print("  Hybrid: ", hybrid.invoke(q)[0].page_content[:50])

Expected outcome: for the first query ("TypeError wrong type"), BM25 and hybrid should outperform pure vector. For "problem accessing object field" (paraphrase of AttributeError), vector should rank AttributeError higher than BM25 does.

Real-world use case

A customer support RAG system for a software product uses hybrid search. Exact error codes (ERR_SSL_PROTOCOL_ERROR) are caught by BM25; natural-language questions like "why does my connection keep dropping?" are caught by vector similarity. Hybrid retrieval reduces "no relevant document found" cases by 30–40% compared to vector-only in production benchmarks.
