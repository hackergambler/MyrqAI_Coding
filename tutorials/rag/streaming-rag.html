Streaming RAG: Real-Time Answer Generation

Waiting for a full RAG response feels slow. Streaming delivers tokens as they're generated — giving users instant feedback. Learn to stream RAG answers with LangChain, the OpenAI SDK, and Server-Sent Events for the web.

In This Lesson

  1. Why Stream RAG Answers?
  2. Streaming with LangChain
  3. Streaming with OpenAI SDK Directly
  4. Serving Streams via FastAPI + SSE
  5. Hands-on Exercise

Why Stream RAG Answers?

A standard RAG call waits until the LLM finishes generating before sending anything back. For a 200-token answer at typical GPT-4o speed (~50 tokens/sec), that's a 4-second blank wait. Streaming fixes this by sending tokens as they arrive, making the UI feel instant.

  • Perceived latency drops — users see the first word in <1 second instead of waiting 4+ seconds
  • Early exit — users can interrupt generation if the answer is already clear
  • Live progress — ideal for long answers like summaries or step-by-step explanations
ℹ️ Retrieval is not streamed

Only the LLM generation step can be streamed. The retrieval step (embedding + vector search) still runs synchronously upfront — typically in 100–300 ms — before the first token is sent.

Streaming with LangChain

LangChain's LCEL (LangChain Expression Language) chains expose a .stream() method that yields chunks as the LLM generates them. Enabling it requires no special setup — just swap .invoke() for .stream().

bash — install dependencies
pip install langchain langchain-openai langchain-community faiss-cpu
Python — streaming RAG with LangChain LCEL
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser
import os

os.environ["OPENAI_API_KEY"] = "your-key"

# Build a small vector store
docs = [
    "Python's asyncio library enables single-threaded concurrent I/O.",
    "FastAPI is built on top of Starlette and uses asyncio natively.",
    "Server-Sent Events (SSE) push updates from server to browser over HTTP.",
    "LangChain's .stream() method yields AIMessageChunk objects.",
]
vectorstore = FAISS.from_texts(docs, OpenAIEmbeddings())
retriever = vectorstore.as_retriever(search_kwargs={"k": 2})

# Build the RAG prompt
prompt = ChatPromptTemplate.from_template(
    """Answer the question using only the context below.

Context: {context}

Question: {question}
Answer:"""
)

# LCEL chain
rag_chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | ChatOpenAI(model="gpt-4o-mini", temperature=0)
    | StrOutputParser()
)

# Stream tokens to stdout
print("Answer: ", end="", flush=True)
for chunk in rag_chain.stream("What is FastAPI built on?"):
    print(chunk, end="", flush=True)
print()  # newline after done
💡 StrOutputParser is required

Without StrOutputParser(), .stream() yields AIMessageChunk objects. The parser extracts the .content string automatically, giving you clean token strings.

Streaming with OpenAI SDK Directly

If you're not using LangChain, the OpenAI SDK's stream=True flag gives the same effect. You retrieve context yourself, then pass it to a streaming chat completion.

Python — streaming RAG with raw OpenAI SDK
from openai import OpenAI
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
import os

os.environ["OPENAI_API_KEY"] = "your-key"

client = OpenAI()
embeddings = OpenAIEmbeddings()

# Build vector store (same as before)
docs = [
    "Python's asyncio library enables single-threaded concurrent I/O.",
    "FastAPI is built on top of Starlette and uses asyncio natively.",
    "Server-Sent Events (SSE) push updates from server to browser over HTTP.",
]
vectorstore = FAISS.from_texts(docs, embeddings)

def stream_rag(question: str):
    # 1. Retrieve relevant chunks
    results = vectorstore.similarity_search(question, k=2)
    context = "\n".join(r.page_content for r in results)

    # 2. Stream generation with context injected
    stream = client.chat.completions.create(
        model="gpt-4o-mini",
        stream=True,
        messages=[
            {"role": "system", "content": "Answer only from the provided context."},
            {"role": "user", "content": f"Context:\n{context}\n\nQuestion: {question}"},
        ],
    )

    # 3. Yield each token chunk
    for chunk in stream:
        delta = chunk.choices[0].delta.content
        if delta:
            yield delta

# Print streaming output
for token in stream_rag("What is SSE?"):
    print(token, end="", flush=True)
print()

Serving Streams via FastAPI + SSE

To stream RAG answers to a web browser, use Server-Sent Events (SSE) — a simple HTTP protocol where the server pushes newline-delimited data: frames. FastAPI's StreamingResponse makes this trivial.

bash — install FastAPI + Uvicorn
pip install fastapi uvicorn
Python — FastAPI streaming RAG endpoint
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from openai import OpenAI
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
import asyncio, os

os.environ["OPENAI_API_KEY"] = "your-key"

app = FastAPI()
client = OpenAI()

# Build vector store at startup
docs = [
    "Python's asyncio library enables single-threaded concurrent I/O.",
    "FastAPI is built on top of Starlette and uses asyncio natively.",
    "Server-Sent Events push updates from server to browser over HTTP.",
]
vectorstore = FAISS.from_texts(docs, OpenAIEmbeddings())

async def rag_stream_generator(question: str):
    # Retrieve synchronously (fast, ~100ms)
    results = vectorstore.similarity_search(question, k=2)
    context = "\n".join(r.page_content for r in results)

    # SSE format: each frame is "data: <token>\n\n"
    stream = client.chat.completions.create(
        model="gpt-4o-mini",
        stream=True,
        messages=[
            {"role": "system", "content": "Answer only from context."},
            {"role": "user", "content": f"Context:\n{context}\n\nQuestion: {question}"},
        ],
    )
    for chunk in stream:
        token = chunk.choices[0].delta.content
        if token:
            yield f"data: {token}\n\n"
            await asyncio.sleep(0)  # yield control to event loop
    yield "data: [DONE]\n\n"

@app.get("/rag/stream")
async def stream_answer(q: str):
    return StreamingResponse(
        rag_stream_generator(q),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"},
    )

# Run: uvicorn app:app --reload
# Test: curl "http://localhost:8000/rag/stream?q=What+is+FastAPI"
⚠️ Disable proxy buffering

Nginx and other reverse proxies buffer responses by default, which defeats streaming. Set X-Accel-Buffering: no in the response headers (shown above) and add proxy_buffering off; in your Nginx config.

Consuming SSE in the browser

JavaScript — reading the SSE stream
const source = new EventSource("/rag/stream?q=What+is+FastAPI");
const output = document.getElementById("answer");

source.addEventListener("message", (e) => {
  if (e.data === "[DONE]") { source.close(); return; }
  output.textContent += e.data;  // append each token
});

source.addEventListener("error", () => source.close());

Hands-on Exercise

Build a streaming Q&A endpoint over a small product FAQ and measure time-to-first-token (TTFT) vs. a non-streaming call.

Python — exercise: measure TTFT for streaming vs blocking
import time
from openai import OpenAI
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import FAISS
import os

os.environ["OPENAI_API_KEY"] = "your-key"

client = OpenAI()
faq = [
    "Refunds are processed within 5 business days.",
    "Premium plan includes unlimited API calls and priority support.",
    "Free tier is limited to 100 requests per day.",
    "You can cancel your subscription anytime from account settings.",
]
vs = FAISS.from_texts(faq, OpenAIEmbeddings())

def build_messages(q):
    ctx = "\n".join(r.page_content for r in vs.similarity_search(q, k=2))
    return [
        {"role": "system", "content": "Answer from context only."},
        {"role": "user", "content": f"Context:\n{ctx}\n\nQuestion: {q}"},
    ]

question = "How long do refunds take?"

# --- Blocking (no stream) ---
t0 = time.perf_counter()
resp = client.chat.completions.create(model="gpt-4o-mini", messages=build_messages(question))
blocking_latency = time.perf_counter() - t0
print(f"Blocking latency (full response): {blocking_latency:.2f}s")
print(f"Answer: {resp.choices[0].message.content}\n")

# --- Streaming (time to first token) ---
t0 = time.perf_counter()
first_token_time = None
full_answer = ""
for chunk in client.chat.completions.create(
    model="gpt-4o-mini", stream=True, messages=build_messages(question)
):
    token = chunk.choices[0].delta.content
    if token:
        if first_token_time is None:
            first_token_time = time.perf_counter() - t0
        full_answer += token
streaming_latency = time.perf_counter() - t0
print(f"Streaming TTFT: {first_token_time:.2f}s")
print(f"Streaming total: {streaming_latency:.2f}s")
print(f"Answer: {full_answer}")

Expected outcome: Blocking latency and streaming total latency will be similar, but TTFT for streaming should be 5–10× lower — typically under 0.5 seconds. This is the improvement users actually perceive.

Real-world use case

A customer-support chatbot at a SaaS company switched from blocking to streaming RAG. Their P50 time-to-first-token dropped from 3.8 seconds to 0.4 seconds. User satisfaction scores improved by 22% — not because answers were more accurate, but simply because users saw the bot start responding immediately rather than showing a loading spinner for 4 seconds.
