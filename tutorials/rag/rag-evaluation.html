RAG Evaluation: RAGAS, Faithfulness & Answer Relevancy

Building a RAG pipeline is only half the work. Measuring whether it actually answers correctly — without hallucinating — is equally critical. Learn the key metrics and how to automate evaluation with RAGAS.

In This Lesson

  1. Why RAG Evaluation Matters
  2. The Four Core RAGAS Metrics
  3. Running RAGAS Evaluation
  4. Evaluating Without Ground Truth
  5. Tracing with LangSmith
  6. Hands-on Exercise

Why RAG Evaluation Matters

A RAG system that retrieves the wrong context or generates an answer not supported by documents is worse than returning "I don't know." Evaluation catches these failures before they reach users. There are two axes to measure:

  • Retrieval quality — did the retriever fetch the right chunks?
  • Generation quality — did the LLM answer faithfully from those chunks?

Without metrics, you are flying blind. A 10% drop in retrieval precision can cut answer quality in half, and you won't notice without measurement.

⚠️ Common mistake

Developers often rely on manual spot-checking ("it looks good"). This misses systematic failures — cases where the model confidently answers incorrectly from irrelevant context. Always run automated evaluation on a representative test set.

The Four Core RAGAS Metrics

RAGAS (Retrieval-Augmented Generation Assessment) defines four complementary metrics that together give a full picture of your pipeline's health.

Faithfulness

Are all claims in the answer supported by the retrieved context? Ranges 0–1. A hallucinated fact that isn't in any retrieved chunk drops this score.

Answer Relevancy

Does the answer actually address the question? Measures whether the response is on-topic, not just factually grounded. Verbose, off-topic answers score lower.

Context Precision

What proportion of retrieved chunks were actually useful? High precision means you're not flooding the LLM with noise.

Context Recall

Were all the chunks needed to answer the question retrieved? Requires a ground-truth answer. Low recall means the retriever missed critical information.

💡 Which metric to optimize first?

Start with Faithfulness. If your LLM is hallucinating, all other metrics are unreliable. Once faithfulness is above 0.85, focus on context precision (reduce chunk noise), then answer relevancy.

Running RAGAS Evaluation

RAGAS evaluates a dataset of (question, answer, contexts, ground_truth) tuples using an LLM judge under the hood. Install it and run your first evaluation in minutes.

bash — install dependencies
pip install ragas langchain-openai datasets
Python — basic RAGAS evaluation
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)
from datasets import Dataset
import os

# Your evaluation dataset: list of dicts
# Each entry needs: question, answer, contexts, ground_truth
data = {
    "question": [
        "What is retrieval-augmented generation?",
        "How does BM25 rank documents?",
    ],
    "answer": [
        "RAG combines a retriever with an LLM to ground answers in documents.",
        "BM25 uses term frequency and inverse document frequency to score relevance.",
    ],
    "contexts": [
        ["Retrieval-Augmented Generation (RAG) fetches relevant documents and passes them to an LLM."],
        ["BM25 is a ranking function based on TF-IDF used in information retrieval."],
    ],
    "ground_truth": [
        "RAG is a technique that retrieves documents and uses them to ground LLM generation.",
        "BM25 ranks documents using term frequency weighted by inverse document frequency.",
    ],
}

dataset = Dataset.from_dict(data)

# Run evaluation (uses GPT-4o as judge by default)
os.environ["OPENAI_API_KEY"] = "your-key"

result = evaluate(
    dataset=dataset,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall],
)

print(result)
# {'faithfulness': 0.96, 'answer_relevancy': 0.91,
#  'context_precision': 1.0, 'context_recall': 0.88}

Interpreting the scores

RAGAS scores range from 0 to 1. A production-grade RAG pipeline typically targets:

  • Faithfulness ≥ 0.85 — less than 15% of claims are hallucinated
  • Answer Relevancy ≥ 0.80 — answers are on-topic
  • Context Precision ≥ 0.75 — retriever avoids noise
  • Context Recall ≥ 0.80 — retriever finds what's needed

Evaluating Without Ground Truth

Ground-truth answers are expensive to create. For production systems, RAGAS supports reference-free evaluation — it can score faithfulness and answer relevancy without a ground truth answer, using the LLM itself as judge.

Python — reference-free evaluation
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy
from datasets import Dataset

# No ground_truth field needed for these two metrics
data_no_gt = {
    "question": ["What is vector search?", "What is chunking in RAG?"],
    "answer": [
        "Vector search finds semantically similar documents using embeddings.",
        "Chunking splits documents into smaller pieces for retrieval.",
    ],
    "contexts": [
        ["Dense vector search encodes text as embeddings and retrieves nearest neighbours."],
        ["Text splitting (chunking) divides long documents into manageable segments."],
    ],
}

dataset = Dataset.from_dict(data_no_gt)
result = evaluate(dataset, metrics=[faithfulness, answer_relevancy])
print(result)
ℹ️ LLM-as-judge cost

RAGAS calls an LLM for each metric per sample. For 100 samples × 4 metrics, expect ~400 LLM calls. Use gpt-4o-mini as the judge to reduce cost by ~10× vs GPT-4o, with only small accuracy loss on standard benchmarks.

Tracing with LangSmith

For ongoing production monitoring, LangSmith traces every RAG call so you can inspect retrievals and answers individually. Combine it with RAGAS for systematic regression testing.

Python — enable LangSmith tracing
import os

# Set these before importing LangChain
os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_API_KEY"] = "your-langsmith-key"
os.environ["LANGCHAIN_PROJECT"] = "rag-evaluation"

# Now all LangChain calls are automatically traced
from langchain_openai import ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings
from langchain.chains import RetrievalQA

# This chain's calls will appear in LangSmith dashboard
vectorstore = FAISS.from_texts(
    ["RAG combines retrieval with generation"],
    OpenAIEmbeddings()
)
qa = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-4o-mini"),
    retriever=vectorstore.as_retriever()
)
answer = qa.invoke("What does RAG do?")
print(answer["result"])

Hands-on Exercise

Build a small evaluation harness that measures faithfulness before and after switching retrieval strategies:

Python — exercise: compare retrieval strategies with RAGAS
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision
from datasets import Dataset
from langchain_community.vectorstores import FAISS
from langchain_community.retrievers import BM25Retriever
from langchain.retrievers import EnsembleRetriever
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain.chains import RetrievalQA
from langchain_core.documents import Document
import os

os.environ["OPENAI_API_KEY"] = "your-key"

corpus = [
    Document(page_content="Python's GIL prevents true multi-threading for CPU-bound tasks"),
    Document(page_content="Use multiprocessing instead of threading for CPU-bound Python work"),
    Document(page_content="asyncio enables concurrent I/O-bound tasks with a single thread"),
    Document(page_content="ThreadPoolExecutor is suitable for I/O-bound workloads in Python"),
]

questions = [
    "Why can't Python threads speed up CPU-bound tasks?",
    "What should I use for concurrent network requests?",
]

def build_qa_chain(retriever):
    return RetrievalQA.from_chain_type(
        llm=ChatOpenAI(model="gpt-4o-mini"),
        retriever=retriever,
        return_source_documents=True
    )

def run_eval(chain, questions, label):
    rows = {"question": [], "answer": [], "contexts": []}
    for q in questions:
        out = chain.invoke(q)
        rows["question"].append(q)
        rows["answer"].append(out["result"])
        rows["contexts"].append([d.page_content for d in out["source_documents"]])
    result = evaluate(Dataset.from_dict(rows), metrics=[faithfulness, answer_relevancy, context_precision])
    print(f"\n{label}: {dict(result)}")

embeddings = OpenAIEmbeddings()

# Strategy 1: vector-only
vs = FAISS.from_documents(corpus, embeddings)
run_eval(build_qa_chain(vs.as_retriever(search_kwargs={"k": 2})), questions, "Vector-only")

# Strategy 2: hybrid (BM25 + vector)
bm25 = BM25Retriever.from_documents(corpus); bm25.k = 2
vec_r = vs.as_retriever(search_kwargs={"k": 2})
hybrid = EnsembleRetriever(retrievers=[bm25, vec_r], weights=[0.4, 0.6])
run_eval(build_qa_chain(hybrid), questions, "Hybrid")

Expected outcome: Hybrid retrieval should score higher on context_precision because it surfaces the most relevant chunks via BM25 for the GIL question, while vector search handles the semantic paraphrase for the second query.

Real-world use case

A legal document Q&A startup evaluated their RAG pipeline with RAGAS and discovered faithfulness was only 0.61 — the LLM was inferring claims not present in retrieved clauses. By switching to a stricter prompt ("Answer ONLY from the provided context. Say 'Not found' if the answer isn't there.") and increasing chunk overlap, faithfulness rose to 0.89 within a week, reducing customer complaints about incorrect legal summaries by 70%.
