Debugging Vector Database Issues

Fix dimension mismatches, embedding model conflicts, slow retrieval, and index corruption in vector stores.

Vector Database Debugging Guide

Vector databases introduce bugs that do not exist in traditional apps: dimension mismatches, embedding model inconsistency, and silent retrieval quality problems.

Bug 1: Dimension Mismatch

❌ InvalidDimensionException: collection has dimension X, got Y
Python - using different models for index vs query
# WRONG: indexed with large (3072 dims), queried with small (1536 dims)
vectorstore = FAISS.from_documents(
    docs, OpenAIEmbeddings(model="text-embedding-3-large")  # 3072 dims
)
vectorstore.save_local("./index")

# Later... different model used to query!
vectorstore = FAISS.load_local(
    "./index",
    OpenAIEmbeddings(model="text-embedding-ada-002"),  # 1536 dims - MISMATCH
    allow_dangerous_deserialization=True
)
results = vectorstore.similarity_search("my query")  # crash or garbage!
✅ One model, stored in metadata alongside the index
Python - single model, saved in meta.json
import json
from pathlib import Path

EMBED_MODEL = "text-embedding-3-small"  # single source of truth
INDEX_DIR = Path("./faiss_index")

def save_index(vs):
    vs.save_local(str(INDEX_DIR))
    (INDEX_DIR / "meta.json").write_text(json.dumps({"model": EMBED_MODEL}))

def load_index():
    meta = json.loads((INDEX_DIR / "meta.json").read_text())
    assert meta["model"] == EMBED_MODEL, f"Model mismatch: {meta['model']} vs {EMBED_MODEL}"
    return FAISS.load_local(str(INDEX_DIR),
        OpenAIEmbeddings(model=EMBED_MODEL),
        allow_dangerous_deserialization=True)

Bug 2: Poor Retrieval Quality

✅ Use hybrid search (semantic + keyword BM25)
Python - hybrid retrieval
from langchain.retrievers import BM25Retriever, EnsembleRetriever

vector_retriever = vectorstore.as_retriever(search_kwargs={"k": 5})
bm25_retriever = BM25Retriever.from_documents(docs)
bm25_retriever.k = 5

hybrid = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.4, 0.6]  # 40% keyword, 60% semantic
)
results = hybrid.invoke("your query here")
print(f"Retrieved {len(results)} documents")

Bug 3: FAISS Index Not Persisted

❌ Index rebuilt from scratch on every restart
Python - broken: in-memory only
# WRONG: index lost on every restart
vectorstore = FAISS.from_documents(docs, embeddings)
# app restarts... index is gone, rebuilt expensively
✅ Save and load FAISS index to disk
Python - persistent FAISS
if os.path.exists("./faiss_index"):
    vectorstore = FAISS.load_local("./faiss_index", embeddings,
        allow_dangerous_deserialization=True)
    print("Loaded existing index")
else:
    vectorstore = FAISS.from_documents(docs, embeddings)
    vectorstore.save_local("./faiss_index")
    print(f"Built index with {vectorstore.index.ntotal} vectors")

# Add new docs incrementally
vectorstore.add_documents(new_docs)
vectorstore.save_local("./faiss_index")
💡 Retrieval score threshold

Always set a minimum similarity threshold. If the best score is below 0.5, the document is likely irrelevant - return a 'not found' answer rather than hallucinating.

Bug 4: Chroma Collection Persists Stale Data

❌ Broken — re-adding documents creates duplicates
Python
import chromadb

client = chromadb.PersistentClient(path="./chroma_db")
collection = client.get_or_create_collection("docs")

# WRONG: run this twice and you get duplicate documents!
collection.add(
    documents=["Python is great", "LLMs are transforming software"],
    ids=["doc1", "doc2"]
)
# Second run raises: "IDs must be unique"
✅ Fixed — upsert instead of add, or check before inserting
Python
# Option 1: upsert (add or update if ID exists)
collection.upsert(
    documents=["Python is great", "LLMs are transforming software"],
    ids=["doc1", "doc2"]
)

# Option 2: delete and rebuild collection when refreshing
try:
    client.delete_collection("docs")
except Exception:
    pass  # doesn't exist yet
collection = client.create_collection("docs")
collection.add(documents=[...], ids=[...])

Bug 5: Metadata Filter Syntax Error in Chroma

❌ Broken — incorrect where clause syntax crashes query
Python
# WRONG: direct key=value in where is invalid syntax
results = collection.query(
    query_texts=["Python tutorial"],
    where={"category": "python"},    # wrong syntax!
    n_results=5
)
# ValueError: Expected where operator, got: python
✅ Fixed — use Chroma's operator syntax
Python
# CORRECT: use explicit operator
results = collection.query(
    query_texts=["Python tutorial"],
    where={"category": {"$eq": "python"}},   # explicit $eq operator
    n_results=5
)

# Other operators: $ne, $gt, $gte, $lt, $lte, $in, $nin, $and, $or
results = collection.query(
    query_texts=["advanced topics"],
    where={"$and": [
        {"category": {"$eq": "python"}},
        {"difficulty": {"$in": ["intermediate", "advanced"]}}
    ]},
    n_results=5
)

Bug 6: LangChain FAISS Score Threshold Ignored

❌ Broken — score_threshold silently not applied
Python
# WRONG: score_threshold only works with similarity_search_with_relevance_scores
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"score_threshold": 0.7}  # silently ignored!
)
docs = retriever.invoke("my query")  # returns ALL docs regardless of score
✅ Fixed — use correct search_type for threshold to take effect
Python
# CORRECT: set search_type to "similarity_score_threshold"
retriever = vectorstore.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={
        "score_threshold": 0.7,  # now actually applied
        "k": 5
    }
)
docs = retriever.invoke("my query")
print(f"Retrieved {len(docs)} docs above 0.7 threshold")
