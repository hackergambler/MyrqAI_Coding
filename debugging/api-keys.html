Most Common API Key Errors

401 OpenAI

Incorrect API key provided

~35% of issues
openai.AuthenticationError: Incorrect API key provided: sk-xxxx. You can find your API key at https://platform.openai.com/account/api-keys.

Common Causes

  • API key has extra whitespace or newline characters
  • Using an old/revoked API key
  • Copying key incorrectly (missing characters)
  • Environment variable not set correctly
  • Using wrong API key for different organization

Solution

1

Verify your API key format (should start with sk- for OpenAI):

import os
api_key = os.getenv("OPENAI_API_KEY")
print(f"Key starts with: {api_key[:7] if api_key else 'NOT SET'}")
print(f"Key length: {len(api_key) if api_key else 0}")
2

Strip whitespace when loading the key:

api_key = os.getenv("OPENAI_API_KEY", "").strip()
client = OpenAI(api_key=api_key)
3

Generate a new API key from the dashboard and update your environment:

# .env file
OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

# Load with python-dotenv
from dotenv import load_dotenv
load_dotenv()
401 All

No API key provided

~25% of issues
openai.AuthenticationError: No API key provided. You can set your API key in code using 'openai.api_key = ', or you can set the environment variable OPENAI_API_KEY.

Common Causes

  • Environment variable not exported/set
  • .env file not loaded
  • Running in different shell/environment
  • API key set after client initialization
  • Docker container missing env vars

Solution

1

Check if environment variable is set:

# Terminal
echo $OPENAI_API_KEY

# Python
import os
print(os.environ.get("OPENAI_API_KEY", "NOT SET"))
2

Set the environment variable properly:

# Linux/Mac - add to ~/.bashrc or ~/.zshrc
export OPENAI_API_KEY="sk-xxxx"

# Windows PowerShell
$env:OPENAI_API_KEY="sk-xxxx"

# Windows CMD
set OPENAI_API_KEY=sk-xxxx
3

For Docker, pass env vars explicitly:

# docker-compose.yml
services:
  app:
    environment:
      - OPENAI_API_KEY=${OPENAI_API_KEY}
    
# Or with docker run
docker run -e OPENAI_API_KEY=$OPENAI_API_KEY myapp
429 All

Rate limit exceeded

~20% of issues
openai.RateLimitError: Rate limit reached for gpt-4 in organization org-xxxx on requests per min (RPM): Limit 500, Used 500, Requested 1.

Common Causes

  • Sending too many requests per minute
  • Exceeding token limits per minute
  • New account with lower limits
  • Free tier restrictions
  • No retry logic with exponential backoff

Solution

1

Implement exponential backoff with retries:

import time
from openai import OpenAI, RateLimitError

client = OpenAI()

def call_with_retry(messages, max_retries=5):
    for attempt in range(max_retries):
        try:
            return client.chat.completions.create(
                model="gpt-4",
                messages=messages
            )
        except RateLimitError as e:
            if attempt == max_retries - 1:
                raise
            wait_time = (2 ** attempt) + random.random()
            print(f"Rate limited. Waiting {wait_time:.2f}s...")
            time.sleep(wait_time)
2

Use the tenacity library for cleaner retry logic:

from tenacity import retry, wait_exponential, stop_after_attempt
from openai import RateLimitError

@retry(
    wait=wait_exponential(min=1, max=60),
    stop=stop_after_attempt(6),
    retry=retry_if_exception_type(RateLimitError)
)
def make_request(messages):
    return client.chat.completions.create(
        model="gpt-4",
        messages=messages
    )
3

Request a rate limit increase from the provider's dashboard, or implement request queuing for high-volume applications.

403 All

Permission denied / Insufficient quota

~12% of issues
openai.PermissionDeniedError: You exceeded your current quota, please check your plan and billing details.

Common Causes

  • No payment method on file
  • Billing limit reached
  • Account suspended
  • Model access not granted (e.g., GPT-4 access)
  • Organization-level restrictions

Solution

1

Check your billing status and add a payment method in the API provider's dashboard.

2

Verify model access—some models require approval:

# Check available models
models = client.models.list()
for model in models.data:
    print(model.id)
3

If using an organization, verify you have the correct org ID set:

client = OpenAI(
    api_key=os.getenv("OPENAI_API_KEY"),
    organization=os.getenv("OPENAI_ORG_ID")  # If required
)
401 Anthropic

Anthropic authentication failed

~8% of issues
anthropic.AuthenticationError: Error code: 401 - {'type': 'error', 'error': {'type': 'authentication_error', 'message': 'invalid x-api-key'}}

Common Causes

  • Using OpenAI key format with Anthropic
  • Key not prefixed correctly (should start with sk-ant-)
  • Wrong environment variable name
  • Header format incorrect in raw API calls

Solution

1

Verify Anthropic key format:

import os
key = os.getenv("ANTHROPIC_API_KEY", "")
print(f"Starts with sk-ant-: {key.startswith('sk-ant-')}")
print(f"Length: {len(key)}")  # Should be ~100+ chars
2

Correct SDK initialization:

from anthropic import Anthropic

client = Anthropic(
    api_key=os.getenv("ANTHROPIC_API_KEY")
)

# Or set env var ANTHROPIC_API_KEY and it auto-loads
client = Anthropic()
3

For raw API calls, use the correct header:

headers = {
    "x-api-key": os.getenv("ANTHROPIC_API_KEY"),
    "content-type": "application/json",
    "anthropic-version": "2024-01-01"
}

API Key Best Practices

🔒

Never Commit Keys

Use .gitignore for .env files. Consider using git-secrets or pre-commit hooks to prevent accidental commits.

🔄

Rotate Regularly

Rotate API keys every 90 days. Immediately rotate if you suspect compromise.

🏷️

Use Project Keys

Create separate API keys for each project/environment. Makes rotation and revocation easier.

💰

Set Spending Limits

Configure billing alerts and hard limits in provider dashboards to prevent unexpected charges.

📊

Monitor Usage

Track API usage patterns. Unusual spikes may indicate key compromise or application issues.

🔐

Use Secret Managers

For production, use AWS Secrets Manager, HashiCorp Vault, or similar services instead of env vars.

Bug Fix: API Key Rotation Causes Downtime

Bug: Deleting the old key before deploying the new one causes a 401 gap

When rotating API keys, the wrong order creates downtime: delete old key → all requests fail → deploy new key. The gap between steps 1 and 3 returns 401 for every request.

Bash - dangerous rotation order
# WRONG order:
# Step 1: Delete sk-old-abc in dashboard  ← DOWNTIME STARTS here
# Step 2: Update env var OPENAI_API_KEY=sk-new-xyz
# Step 3: Redeploy app                    ← DOWNTIME ENDS here
# Every request in the gap → 401 Unauthorized

✅ Fix — create new key first, verify it works, THEN revoke old key

Bash - zero-downtime rotation
# CORRECT order (zero downtime):
# Step 1: Create sk-new-xyz in dashboard (both keys active)
# Step 2: Update secret manager / env var to sk-new-xyz
# Step 3: Redeploy all services
# Step 4: Verify health: curl -H "Authorization: Bearer $OPENAI_API_KEY" \
#           https://api.openai.com/v1/models | python3 -c "import sys,json; print('OK' if json.load(sys.stdin).get('data') else 'FAIL')"
# Step 5: NOW revoke sk-old-abc (zero downtime achieved)

Bug Fix: API Key Leaked in Git History

Bug: Hardcoded key committed to git — the most costly mistake in AI development

GitHub bots scan public repos 24/7 for API keys. A leaked key is typically abused within minutes. Simply deleting the file does not help — the key is in git history forever unless removed properly.

Python - never do this
# WRONG: hardcoded key committed to repo
from openai import OpenAI
client = OpenAI(api_key="sk-proj-abc123xyz...")  # leaked!

✅ Fix — env vars + .gitignore + git-secrets pre-commit hook

bash - safe setup
# 1. Never commit .env — add to .gitignore FIRST
echo ".env" >> .gitignore

# 2. Store key in .env file
echo 'OPENAI_API_KEY=sk-proj-abc123' > .env

# 3. Install git-secrets to block commits with secrets
brew install git-secrets  # or: pip install detect-secrets
git secrets --install
git secrets --register-aws  # also catches OpenAI-style keys

# 4. If already committed: revoke immediately on provider dashboard,
#    then use git-filter-repo to purge from history
pip install git-filter-repo
git filter-repo --path .env --invert-paths
