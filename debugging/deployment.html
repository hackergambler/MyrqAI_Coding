Debugging AI App Deployment Issues

Fix deployment errors for AI apps: Docker, Lambda, Vercel, and environment-specific production bugs.

Why AI Apps Fail in Production but Work Locally

AI apps depend on network calls, API keys, large model files, and sometimes GPU drivers. This guide covers the most common production deployment failures.

Bug 1: API Key Not Available in Container

❌ AuthenticationError in Docker but works locally
Dockerfile - broken
# WRONG: hardcoding key in image is a security risk
FROM python:3.12-slim
ENV OPENAI_API_KEY=sk-abc123...  # Never do this!
COPY . .
RUN pip install -r requirements.txt
CMD ["python", "app.py"]
✅ Pass secrets at runtime, never bake into image
bash - correct Docker deployment
# Pass at runtime
docker run -e OPENAI_API_KEY=$OPENAI_API_KEY myapp:latest

# Or use .env file
docker run --env-file .env myapp:latest

# Verify inside container
docker exec -it myapp python -c \
  "import os; print(os.environ.get('OPENAI_API_KEY', 'NOT SET')[:10])"

Bug 2: Lambda Timeout on Long LLM Calls

❌ Task timed out after 29.00 seconds
Python - Lambda hitting default 3s timeout
def handler(event, context):
    client = OpenAI()
    # gpt-4o can take 20-60s for long outputs
    resp = client.chat.completions.create(
        model="gpt-4o",
        max_tokens=4096  # slow + too long for Lambda timeout
    )
    return {"answer": resp.choices[0].message.content}
✅ Use faster model, limit tokens, raise Lambda timeout to 60s
Python - Lambda-friendly approach
def handler(event, context):
    client = OpenAI()
    resp = client.chat.completions.create(
        model="gpt-4o-mini",  # 3-5x faster
        messages=[{"role": "user", "content": event["prompt"]}],
        max_tokens=512  # hard limit
    )
    return {"answer": resp.choices[0].message.content}

# For long jobs: use SQS + async Lambda pattern
# Lambda 1: queue to SQS, return job_id immediately
# Lambda 2: triggered by SQS, runs LLM, stores in DynamoDB
# Client: polls GET /result/{job_id}

Bug 3: Missing Dependencies in Production

❌ ModuleNotFoundError: No module named tiktoken
Python - incomplete requirements.txt
# requirements.txt is missing AI packages
fastapi==0.115.0
uvicorn==0.32.0
# Missing: openai, tiktoken, langchain, anthropic, etc.
✅ Pin all dependencies with pip freeze
bash - generate complete requirements
# Capture ALL installed packages with exact versions
pip freeze > requirements.txt

# Or use pip-compile for cleaner dep resolution
pip install pip-tools
pip-compile requirements.in

# Minimal requirements.in
# openai>=1.52
# anthropic>=0.39
# langchain>=0.3
# fastapi>=0.115
# uvicorn[standard]>=0.32

Bug 4: Cold Start Latency

✅ Initialize client at module level for reuse across Lambda invocations
Python - warm connection pool
from openai import OpenAI
import os

# Module-level init runs ONCE per container, reused every invocation
client = OpenAI(
    api_key=os.environ["OPENAI_API_KEY"],
    max_retries=3,
    timeout=30.0
)

def handler(event, context):
    # Reuses existing client - no cold init cost
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": event["prompt"]}]
    )
    return {"result": resp.choices[0].message.content}
⚠️ CORS in production

If your frontend gets CORS errors, add CORSMiddleware to FastAPI with your specific production domain. Never use allow_origins=["*"] in production.

Bug 5: CORS Error in Production

❌ Broken — missing or wrong CORS headers block frontend
Python
# WRONG: no CORS middleware — browser blocks all requests from frontend
from fastapi import FastAPI
app = FastAPI()

# No CORSMiddleware added!
# Browser error: "Access to fetch ... blocked by CORS policy"
✅ Fixed — add CORSMiddleware with explicit origins
Python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
import os

app = FastAPI()

# Read allowed origins from env (never hardcode in code)
ALLOWED_ORIGINS = os.environ.get("ALLOWED_ORIGINS", "http://localhost:3000").split(",")

app.add_middleware(
    CORSMiddleware,
    allow_origins=ALLOWED_ORIGINS,   # NEVER use ["*"] in production
    allow_credentials=True,
    allow_methods=["GET", "POST"],
    allow_headers=["Authorization", "Content-Type"],
)

Bug 6: Health Check Endpoint Missing — Load Balancer Kills Pods

❌ Broken — no health endpoint, Kubernetes/ECS terminates healthy pods
Python
# WRONG: no /health endpoint
# AWS ALB, Kubernetes, and ECS all probe /health (or /healthz)
# When they get 404, they mark the pod as unhealthy and kill it
app = FastAPI()

@app.post("/chat")
async def chat(body: dict):
    return {"answer": "..."}
✅ Fixed — add health and readiness endpoints
Python
from datetime import datetime
import os

START_TIME = datetime.now()

@app.get("/health")            # liveness: am I running?
async def health():
    return {"status": "ok", "uptime": str(datetime.now() - START_TIME)}

@app.get("/ready")             # readiness: am I ready for traffic?
async def ready():
    # Check your dependencies here
    api_key_set = bool(os.environ.get("OPENAI_API_KEY"))
    if not api_key_set:
        from fastapi import HTTPException
        raise HTTPException(status_code=503, detail="API key not configured")
    return {"status": "ready"}

Bug 7: Docker Image Too Large — Slow Deploys

❌ Broken — single-stage build includes dev tools, gigabytes of waste
Dockerfile
FROM python:3.12       # 1.02GB base image
COPY . .
RUN pip install -r requirements.txt  # installs everything including dev tools
CMD ["uvicorn", "app:app"]
# Result: 2.5GB+ image, 3-minute deploys
✅ Fixed — multi-stage build with slim base
Dockerfile
# Stage 1: build dependencies
FROM python:3.12-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# Stage 2: lean runtime image
FROM python:3.12-slim AS runtime
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY src/ ./src/
ENV PATH=/root/.local/bin:$PATH
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1
USER nobody                          # never run as root
CMD ["uvicorn", "src.app:app", "--host", "0.0.0.0", "--port", "8000"]
# Result: ~200MB image, 30-second deploys
